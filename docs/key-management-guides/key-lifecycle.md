# Key Lifecycle Management Guide

## Overview

A key lifecycle policy defines the full life of a cryptographic key from generation through decommissioning. Gaps in any phase of the lifecycle are a common source of cryptographic security failures.

---

## Phase 1: Key Generation

**Requirements:**
- Use a hardware-based CSPRNG or a verified software CSPRNG (see `policies/crypto-baselines/standard.yaml`)
- Generate keys on a system appropriate for the key's sensitivity tier:
  - **High sensitivity (production secrets, signing keys, customer data encryption):** Air-gapped or HSM-bound generation
  - **Standard sensitivity (API keys, session keys):** Secrets manager with IAM-controlled access
  - **Development/test keys:** Standard development environment; clearly labelled as non-production
- Document the generation event: date, system, entropy source, generated by
- Never generate keys in shared, multi-tenant environments unless key material is immediately wrapped

**Key size minimums:**
- AES: 128-bit minimum; 256-bit recommended
- RSA: 2048-bit minimum; 3072-bit for keys expected to be in use beyond 2030
- EC: P-256 minimum; X25519/Ed25519 for new systems
- See `policies/crypto-baselines/standard.yaml` for full reference

---

## Phase 2: Key Registration and Inventory

**Requirements:**
- Register every cryptographic key in the key management inventory with:
  - Key ID or alias
  - Key type and algorithm
  - Purpose (data encryption, API authentication, signing, etc.)
  - Creator and creation date
  - Owning team / service
  - Storage location
  - Rotation policy
  - Expiry or planned decommission date

**Why this matters:**
Untracked keys cannot be rotated, audited, or revoked. "Shadow keys" — keys created outside the inventory process — are a frequent source of incident escalation.

---

## Phase 3: Key Distribution

**Requirements:**
- Key material must travel only over encrypted, authenticated channels
- Never transmit raw key material via email, Slack, Jira, or other collaboration tools
- For service consumption: use secrets manager injection at runtime (not at build time)
- For human access: use short-lived credentials with JIT (Just-In-Time) access where possible
- Log all key access events: who, when, purpose

**Patterns to avoid:**
- Hardcoding keys in application code or Dockerfile
- Storing keys in CI/CD environment variables visible to all pipelines
- Sending keys as plaintext in internal communications

---

## Phase 4: Key Usage and Monitoring

**Requirements:**
- Monitor key usage for anomalies:
  - Unusual access patterns (off-hours, unexpected systems)
  - High access volume (potential key misuse or scraping)
  - Access from unexpected principals
- Alert on:
  - First use of a newly generated key
  - Access to keys for decommissioned services
  - Key access from IPs outside expected ranges
- Implement purpose limitation: a key created for data encryption should not be used for signing

---

## Phase 5: Key Rotation

**Rotation policy by key type:**

| Key Type | Recommended Interval | Trigger for Immediate Rotation |
|---|---|---|
| API keys | 90 days | Suspected compromise, employee departure |
| Session/auth tokens | 24 hours – 7 days | Any suspected compromise |
| Data encryption keys (DEK) | 1 year | Data breach, DEK compromise |
| Key encryption keys (KEK) | 2 years | DEK compromise, access policy change |
| TLS certificates | Per validity period | Private key compromise, CA compromise |
| Signing keys (code/data) | 1–2 years | Private key compromise |
| Cold storage keys | Per custody policy | Any suspected compromise |

**Rotation process:**
1. Generate new key material (new key ID — do not overwrite)
2. Update all consumers to use the new key (deploy, test, verify)
3. Mark the old key as deprecated — it should only decrypt existing data, not encrypt new
4. After a transition period, decommission the old key
5. Update the key inventory

---

## Phase 6: Key Decommissioning

**Requirements:**
- Before decommissioning:
  - Confirm all data encrypted with this key has been re-encrypted with the replacement key
  - Confirm no services depend on the key
  - Confirm the key is not in active rotation (not the current "encrypt" key, only the "decrypt" key)
- Decommissioning steps:
  1. Mark key as INACTIVE in the secrets manager / key management system
  2. Remove key from all consumer configurations
  3. Wait for propagation and confirm no new usage
  4. Mark key as DELETED / DESTROYED
  5. For HSM-bound keys: execute key ceremony to formally destroy key material
  6. Document decommission: who authorized it, when, reason

**Key destruction:**
- For software keys: overwrite key material before deallocation
- For HSM keys: invoke the HSM's key destruction mechanism
- For physical copies (paper backups): shred or burn
- Document the destruction method and witness

---

## Audit and Compliance

- Conduct a quarterly key inventory audit: confirm all keys in the inventory are accounted for
- Annual review of rotation policies against current NIST/IETF guidance
- Include key management controls in SOC 2 / ISO 27001 audit scope
- Retain key usage logs per your retention policy (typically 1–3 years)
